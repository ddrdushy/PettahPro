# Inventory Module Design Spec — Multi-Tenant Accounting SaaS (Sri Lanka)

> Specification for the Inventory Management module. Ties directly into Accounting module (perpetual COGS, per-category valuation). Target market: **Sri Lanka only**. Scope: **full system, not MVP**.

---

## 1. Scope & Principles

- **Perpetual inventory model** — every stock movement posts to the ledger in real time (locked in Accounting spec)
- **Per-item-category valuation**: WAVG / FIFO / Specific / Standard Cost (locked in Accounting spec)
- **Warehouse = independent entity**, flexibly mapped to branches (locked in Layer 2 spec)
- **Minimal-entry / Easy Mode** is a platform-wide UX principle (section 3)
- **No live bank integrations, no GPU, no voice** (locked platform-wide)
- **Manufacturing / full BOM deferred** to a Phase 2 module; lightweight assembly via Bundle item type is supported here

## 2. Item Master

### 2.1 Item types

| Type | Stock tracked | Notes |
|---|---|---|
| **Stock item** | Yes | Physical inventory that moves in/out |
| **Service** | No | Non-stock billable (consultation, delivery, repair labour) |
| **Bundle / Kit** | Yes (components) | Static component list; sale consumes components automatically. No routing, no labour. See section 9. |
| **Non-inventory** | No | One-off custom items, straight to GL on transaction |
| **Custom** | Tenant-defined | Tenant can add their own item types with tenant-specific behavior |

**Fixed assets** live in the Accounting module (separate master), not here.

### 2.2 Identity & descriptive fields

- **Item code** — auto-generated by default; tenant configures prefix/pattern per category (e.g. `TEX-` for textile, `PHR-` for pharmacy). Manual override allowed.
- **Name** — required; multi-language support (EN/TA/SI). Tamil name for Pettah staff, Sinhala for Kandy, etc.
- **Description** — optional long text
- **Category** — unlimited-depth hierarchy tree (see section 2.4)
- **Brand** — optional
- **HS Code** — for items subject to import duty / customs
- **Barcodes** — multiple barcodes per item (supplier's GTIN + our internal code + promo barcode)
- **Tags** — free-form user tags for reporting (distinct from the single accounting Tag dimension)
- **Valuation method** — inherited from category; per-item override allowed

### 2.3 Minimal-entry item creation (locked design pattern)

**Minimum required on create**: Name + Primary UoM. Everything else optional, added progressively.

**LLM-assist capture paths**:
- **Barcode scan** → public GTIN/UPC database lookup + our crowdsourced tenant data → auto-fill name, brand, category, unit size → user confirms
- **Product label photo** → Tesseract OCR + LLM parser → extract name, net weight/volume, HS code, expiry format → user confirms
- **Category auto-suggestion** from name (LLM classifies into tenant's category tree)
- **Multi-language name translation** — type English, LLM suggests Tamil + Sinhala → user edits if needed
- **CSV bulk upload** — same LLM-assist pattern at row level during import

**Rule**: all LLM-filled fields highlighted yellow until user reviews. Never silently commit LLM output.

### 2.4 Category tree

- Unlimited-depth hierarchy (Textile → Cotton → Cotton Shirting → 100% Cotton / Cotton Blend)
- Breadcrumb display in reports
- Each category can carry defaults: valuation method, tax code, code prefix, reorder defaults
- Child categories inherit from parent unless overridden

### 2.5 Units of Measure (UoM)

**Primary UoM + alternate UoMs with conversion factors**:
- Item has one Primary UoM (stock is tracked in this unit)
- Additional UoMs defined with conversion factor to Primary (1 box = 10 strips = 100 tablets)
- Purchase/sale can happen in any defined UoM; system converts to Primary for stock ledger
- Pricing maintained per UoM (buy by box price, sell by tablet price)

### 2.6 Item variants (parent-child structure)

- Parent item holds common info (name, HS code, supplier, base price tier)
- Variants defined by tenant-configurable attribute axes (Size, Color, Material, Style, etc. per item)
- Variants inherit from parent; override where different
- Stock, sales, pricing tracked per variant
- Reports can roll up to parent or drill to variant

Example: shirt (parent) × Size (S/M/L/XL) × Color (White/Blue/Black) = 12 variants with shared parent data

### 2.7 Batch / Serial / Expiry tracking

Three independently configurable per item:

| Capability | Configured per item | Use case |
|---|---|---|
| **Batch tracking** | Toggle | Pharmacy, food, paint — stock per batch, FIFO by batch |
| **Serial tracking** | Toggle | Electronics, vehicles, high-value — unique serial per unit; full trace from purchase to sale |
| **Expiry tracking** | Toggle | Perishables — expiry date per lot; FIFO by expiry; expiry alerts |

Combinations supported: pharmacy item = batch + expiry; electronics = serial only; textile = neither.

**Recall capability**: batch-level invoice trace — if batch has quality issue, system lists every invoice that included it.
**Warranty**: tracked per serial (electronics, vehicles).

### 2.8 Pricing

**Resolution hierarchy** on any sale:
1. **Customer-specific override** (highest priority — negotiated prices for key customers)
2. **Customer's assigned price list** (Retail / Wholesale / Dealer / VIP — tenant defines)
3. **Default price list**

**Modifiers layered on top**:
- **Volume breaks** — tier pricing (1–10 @ LKR 100, 11–50 @ LKR 95, 51+ @ LKR 90)
- **Time-bound promos** — LKR 80 from Dec 1 to Dec 31
- Applied automatically when thresholds met; user sees calculated price with breakdown

**Price list management**:
- Multiple named price lists per tenant
- Customer → price list assignment (default or mandatory)
- CSV bulk update — upload SKU + new price + effective date
- Bulk % adjustment — "apply 5% increase to all textile items effective March 1"
- Preview before commit

## 3. Minimal-Entry / Easy Mode (platform-level principle)

Extended from Layer 2; formalized as platform-wide UX standard here.

### 3.1 Principle — every entry screen ships in two modes

- **Easy Mode (default)** — minimum required fields visible, everything else hidden behind *"show more"*. Smart defaults for 80% of cases. Rapid entry.
- **Advanced Mode** — full field set exposed, for power users (accountant, CA, stock controller doing monthly review).

Toggle is user-preference per-screen, remembered between sessions.

### 3.2 Concrete application per Inventory screen

**Item creation (Easy Mode)**
- Required: Name + Primary UoM only
- Everything else optional, progressively filled
- LLM-assist paths as specified in 2.3

**Purchase GRN (Easy Mode)**
- Required: Supplier + date + items with qty
- LLM-assist: photo/PDF of supplier invoice → Tesseract parser → draft GRN with pre-filled fields
- Unknown SKUs flagged; user maps or creates with LLM help
- Recent suppliers + recent items ranked by last-used

**Sales invoice (Easy Mode)**
- Required: Customer + items with qty
- Auto-derived: price (from hierarchy), tax (from item tax code), date, invoice#
- Barcode scanning at line level — scan, qty defaults 1, next scan
- Keyboard arrow edits last qty

**Stock adjustment (Easy Mode)**
- Required: Warehouse + item + qty variance + reason code
- Auto-defaults: date (today), branch
- Bulk mode: CSV paste for multi-item adjustments

**Stock count (Easy Mode)**
- Mobile-first: barcode scan + qty
- Batch scan mode — scan 20 items, enter each qty once, commit batch
- Offline capable with sync-on-reconnect

**Transfers (Easy Mode)**
- Required: From warehouse + To warehouse + items + qty
- Multi-line quick entry via barcode scan

**Price list update (Easy Mode)**
- CSV upload with LLM column mapping
- Bulk % adjustment by category/filter
- Preview before commit

### 3.3 Universal helpers across all screens

- **Keyboard-first**: Alt+S save, Alt+N new, Alt+P post, Tab order logical, Enter advances
- **Recent + frequent suggestions**: dropdowns rank by recent use for current user
- **Sticky defaults**: last-used warehouse, tax code, payment mode persist
- **Inline validation**: errors under field, never modal
- **Progressive save**: drafts auto-saved every N seconds
- **Undo last action**: 30-second undo window for irreversible-feeling actions (e.g. Post Invoice)

## 4. Stock Movements

### 4.1 Movement types (all in scope)

**Inward**
- Purchase GRN (goods received from supplier)
- Sales return (customer returns goods)
- Stock adjustment — positive
- Opening balance entry
- Transfer in (from another warehouse)
- Bundle component addition (reverse of sale, if bundle return)

**Outward**
- Sales invoice / delivery
- Purchase return (we return to supplier)
- Stock adjustment — negative
- Transfer out
- Internal consumption (staff uniforms, promotional giveaways, samples)
- Bundle component consumption (on bundle sale)

**Non-affecting on total stock**
- Warehouse transfer (shifts location only)
- Stock count variance reporting (posts as adjustment after count)

### 4.2 Warehouse transfers — two-step workflow

- User creates Transfer Request → from/to warehouses, items, qty
- **Dispatch step**: source warehouse confirms → qty leaves source, enters virtual "in-transit" location
- **Receive step**: destination warehouse confirms → qty leaves in-transit, enters destination
- Gap = stock "in transit", not available at either end, visible in reports
- **Discrepancy handling**: if dispatch was 100 but receipt was 95, shortage posted + investigation flag
- **Permissions**: only roles with write access to BOTH warehouses can initiate; source confirms dispatch; destination confirms receipt
- **No GL impact**: same tenant, same books — stock ledger only
- **Valuation**: transferred stock carries its source cost (no markup)

### 4.3 Stock adjustments

- **Reason code mandatory** — pre-populated SL-typical (damage, theft, expiry, shrinkage, sample, miscount, system error, other) + tenant-configurable
- **Positive adjustment**: Dr Inventory, Cr *Stock Gain*
- **Negative adjustment**: Dr *Stock Loss* (sub-analyzed by reason), Cr Inventory
- **Approval threshold**: adjustments above tenant-set qty or LKR value → Accountant or Owner approval
- **Supporting documents**: photo upload for damage, report upload for theft (retrievable later for insurance claims)
- **Adjustment register report** by reason code and period

### 4.4 Physical stock count (simpler version)

- **Count setup**: scope = full warehouse / category / ABC class / specific items
- **Count execution**: mobile app + barcode scan (preferred) OR printed count sheet + manual entry. Blind count (counter cannot see system qty).
- **Variance review**: System Qty vs Counted Qty vs Variance (qty + LKR). User classifies each variance with reason code.
- **Adjustment posting**: once approved, variances posted as batch adjustment entry with reason codes. Count becomes immutable audit record.

**Skipped** (can be done manually if tenant wants): double-count workflow for high-value items, automated chronic-variance pattern tracking, elaborate multi-counter recount.

### 4.5 Reorder management (alert-only)

**Per item, per warehouse configuration**:
- Reorder point (when stock ≤ X, alert)
- Reorder quantity (EOQ)
- Lead time in days
- Safety stock level
- Stockout alert trigger (zero stock)

**Alert mechanisms**:
- Dashboard tile on Owner / Stock Keeper dashboard
- Email / SMS digest (daily or real-time, configurable per user)

**Not in scope**: auto-draft PO generation. Tenant creates POs manually from alerts.

### 4.6 Negative stock handling

- **Default**: hard-block sale when stock insufficient
- **Override**: Owner or Accountant can permit negative stock with reason logged
- **No per-item-category rule** — single global behavior with permission gate

## 5. Valuation & Costing

### 5.1 Per-category valuation method

Tenant configures per category:
- **Weighted Average (WAVG)** — default for most SL shops
- **FIFO** — pharmacy, food, perishables (works with batch tracking)
- **Specific Identification** — serial-numbered items
- **Standard Cost** — manufacturing-style shops
- **LIFO** — not supported (prohibited under LKAS)

### 5.2 Valuation mechanics

| Method | On purchase | On sale |
|---|---|---|
| WAVG | New avg = (old qty × old cost + new qty × new cost) / total qty | COGS = current WAVG × sold qty |
| FIFO | New lot added to queue with own cost | COGS draws from oldest lot; queue advances |
| Specific | Each unit keeps its purchase cost | COGS = exact cost of specified unit |
| Standard | Difference to standard posted to Purchase Price Variance | COGS = standard cost |

### 5.3 Standard cost revision

- Tenant updates standard cost → **existing stock revalued at new standard**
- Difference posted to *Inventory Revaluation Gain/Loss* account
- Revision logged with user + reason

### 5.4 Landed cost

**Components captured on GRN / Bill**:
- Freight (domestic + international)
- Insurance
- Customs duty
- Clearing / forwarding fees
- Loading / unloading
- Other (custom reason codes)

**Allocation method (user picks per GRN)**:
- Pro-rata by value
- Pro-rata by quantity
- Pro-rata by weight (requires weight per item)
- Manual per-line split

**Accounting treatment**:
- Freight, insurance, customs, clearing → capitalized to inventory cost
- FX gain/loss on supplier payment → separate FX account, not landed cost

**Post-GRN bill linkage**:
- Clearing agent invoices often arrive weeks later
- Post GRN first with estimated landed cost
- Later link actual supplier bills → item cost adjusted retrospectively
- Difference posted to *Landed Cost Variance*
- **Lock window**: GRN landed cost editable for N days (tenant-configurable) OR until first related sale posts, whichever earlier

### 5.5 Sales freight / packing charges

**Bundled into item price** — no separate revenue line on invoice. Cleaner invoice layout; freight/packing built into per-unit pricing as needed. Simpler accounting, no per-invoice toggle complexity.

### 5.6 In-transit / bonded warehouse

**Skipped**. Stock tracked only once in a real warehouse. Importer workflow handled via standard Purchase → GRN → stock-in on physical receipt.

## 6. Warehouse Model

### 6.1 Warehouse as single unit

No internal bin/rack/zone hierarchy. Each warehouse is a single addressable location. Keeps data model simple; fits SL SME reality.

### 6.2 Branch ↔ Warehouse mapping

Inherited from Layer 2:
- **Tied** — warehouse serves one branch only
- **Central** — one warehouse serves many branches
- **Mixed** — central + local warehouses

Defined via `branch_warehouse_access` join with read / write / transfer permissions.

## 7. Supplier & Customer Linkages

### 7.1 Supplier-item linkage (multi-supplier per item)

Per supplier-item combination:
- Supplier's part number (differs from our SKU)
- Supplier's price
- Lead time in days
- MOQ (minimum order quantity)
- Preferred flag (default supplier on PO creation)
- Last purchased date + last price
- Full purchase price history for negotiation

### 7.2 Customer-item linkage

- **Customer buy history** — what, at what price, when, quantity
- **Suggested reorder** — *"Customer usually orders monthly, last order 28 days ago — suggest this basket?"*
- **Preferred substitutes** — *"When Item A is out of stock, this customer accepts Item B"*
- **Standing orders / recurring orders** — auto-generate drafts on schedule (e.g. every Monday customer gets this basket)

---

## 8. Returns

### 8.1 Customer returns (simplified)

- Sales Return linked to original invoice (must exist in system)
- System pre-fills items + quantities; user adjusts what's actually returning
- **Stock restored to sellable inventory** by default (no condition classification in this phase)
- **Accounting**: Credit Note posted (reverses AR), tax reversal on returned portion
- **Refund**: credit to customer account (offset future) OR refund cash/bank (user choice)

**Not in scope**: restocking fees, return-window approval thresholds, damage vs restock condition classification. Can extend later.

### 8.2 Supplier returns (Purchase Return / Debit Note)

- Linked to original Purchase Bill / GRN
- **Reason**: Quality reject / Excess delivery / Wrong item / Damaged in transit
- **Accounting**: Debit Note posted (reduces AP), stock reduced, Input VAT reversed
- **Settlement**: offset against future supplier bill OR supplier refund (rare in SL)

---

## 9. Bundle / Kit (lightweight, no BOM)

- Bundle item = static list of component items + quantities
- Examples: gift basket (pen + notebook + chocolate), pharmacy combo pack (Panadol + vitamin C + cough syrup), "buy 1 get 1"
- **On sale of bundle**:
    - Bundle SKU reduces by 1
    - Each component reduces by its defined quantity
    - Single invoice line (bundle name + price), or option to show components as sub-lines
- **Valuation**: bundle cost = sum of component costs at time of sale (using each component's valuation method)
- **Pricing**: bundle has its own price (often less than sum of components — the promotional angle)

**No** routing, no labour, no work orders, no manufacturing variance. Full manufacturing capability deferred to Phase 2.

## 10. Reports (full scope)

### 10.1 Stock status
- Stock on hand (by item, warehouse, batch/serial/expiry)

### 10.2 Movement
- Item movement ledger (all in/out for a SKU over period)
- Warehouse ledger (all movements for a warehouse)

### 10.3 Analytics
- **ABC classification** — auto-computed on rolling 12-month window; A = top 20% by value, B = next 30%, C = bottom 50%

- Reorder alerts / below-reorder-point list
- Expiry report (items expiring in N days)

- Stock count history + variance trends

### 10.5 Compliance / audit
- Physical stock count certificate (auditor format)
- Adjustment register by reason code
- Transfer register

### 10.6 Custom report builder
- Drag-drop field selection
- Filter builder (item, warehouse, date, tag, supplier, customer)
- Saved views per user
- Schedulable email delivery

## 11. POS Integration

- Every POS sale → instant stock decrement, perpetual COGS posted
- Every POS refund → instant stock increment, COGS reversal
- Every POS void → reverses the original posting
- **Offline POS**: local queue, syncs to central when connection restored; POS is source of truth for sold qty; system reconciles on sync
- **End-of-day Z-report** per POS terminal — sales summary reconciles with stock movements
- **Peripherals supported as standard**: cash drawer, receipt printer (thermal + laser), customer display, barcode scanner, scale integration (for deli / fresh / by-weight items)

## 12. Numbering Series

- **Prefix configurable** per doc type per branch (e.g. `TEX-INV-2026-`, `GRN-PTH-`)
- **Sequence reset**: annual / monthly / never — tenant configures
- **Format**: year/month inclusion, padding length (4/6 digits) — tenant configures
- **Gaps**: allow (voided transactions keep number, marked void, never reused) or no-gap (renumber) — tenant configures
- **SL-audit defaults on setup**: annual reset + year in number + gaps allowed with void trail

## 15. Accounting Integration — GL Posting Map

Every stock movement posts to the ledger. Full mapping:

| Movement | Debit | Credit |
|---|---|---|
| Purchase GRN | Inventory + Input VAT | Supplier AP |
| Purchase Return | Supplier AP | Inventory + Input VAT |
| Sales Invoice | Customer AR | Sales Revenue + Output VAT; also COGS / Inventory |
| Sales Return | Sales Revenue + Output VAT | Customer AR; also Inventory / COGS |
| Stock Adjustment + | Inventory | Stock Gain |
| Stock Adjustment − | Stock Loss (by reason) | Inventory |
| Warehouse Transfer | — (stock ledger only) | — (no GL impact) |
| Stock Count Variance | Same as Adjustment | Same as Adjustment |
| Internal Consumption | Expense (by purpose) | Inventory |
| Opening Balance | Inventory | Opening Balance Equity |
| Landed Cost Capitalization | Inventory | Supplier AP / Clearing Agent AP |
| Landed Cost Variance | Inventory or Landed Cost Variance | Landed Cost Variance or Inventory |
| Bundle Sale | Customer AR | Sales Revenue + Output VAT; COGS per-component / Inventory per-component |
| Standard Cost PPV | Purchase Price Variance | Supplier AP (offset to Inventory for standard portion) |

## 17. Permissions & Approvals

All roles, permissions, and approval chains persist in the `roles` / `role_permissions` / `user_roles` tables (`data-model-02-identity.md §4`). This section enumerates the inventory-specific permission keys, scope rules, and default approval gates.

### 17.1 Roles that interact with Inventory (defaults; tenants can customize)

| Role | Default scope | Inventory capabilities |
|---|---|---|
| **Owner / Finance Head** | Tenant-wide | Full: item master, costing method, landed cost, adjustments without cap, negative stock override, reorder config, warehouse access rules. |
| **Inventory Manager** | Tenant-wide | Item master CRUD, transfer approvals, adjustments within cap, stock counts, reorder tuning. Cannot change costing method or delete items with history. |
| **Stock Keeper** | Warehouse(s) assigned | Receive (GRN), dispatch (DN/transfer), execute adjustments within cap, stock count execution. Cannot edit item master price/cost. |
| **Accountant** | Tenant-wide (read + close) | Read valuation reports, approve standard cost revisions, approve landed cost postings, month-end inventory close. |
| **Salesperson / Cashier** | Branch | Read stock availability for sellable items; no write permissions. |
| **Purchaser (Buy module link)** | Tenant-wide | Read item master + supplier linkages; update preferred supplier and reorder configs; no stock movement rights. |
| **External Auditor** | Tenant-wide (read) | Read-only on ledger, adjustments, counts, valuation. Time-bounded. |

### 17.2 Permission keys (`role_permissions.module = 'inventory'`)

| Action key | Typical grantees | `conditions_json` |
|---|---|---|
| `inventory.view_items` | All operational roles | — |
| `inventory.create_item` | Owner, Inventory Manager | — |
| `inventory.edit_item_master` | Owner, Inventory Manager | Price/cost fields may require approval — see §17.3 |
| `inventory.edit_costing_method` | Owner | Locked after first posting — revaluation required |
| `inventory.view_cost_and_margin` | Owner, Inventory Manager, Accountant | Hidden for Salesperson/Cashier by default |
| `inventory.transfer_initiate` | Stock Keeper, Inventory Manager | `{"scope":"warehouse","warehouse_ids":[...]}` — source must be in scope |
| `inventory.transfer_receive` | Stock Keeper (destination) | Destination warehouse in scope |
| `inventory.adjust_stock` | Stock Keeper, Inventory Manager, Owner | `{"max_adjust_value_lkr": 50000}` — Stock Keeper; `{"max_adjust_value_lkr": 500000}` — Inventory Manager; uncapped — Owner. Reason code mandatory. |
| `inventory.override_negative_stock` | Owner, Inventory Manager | `{"reason_required": true}` — flag + audit always; tenant toggle may disable entirely |
| `inventory.run_stock_count` | Stock Keeper, Inventory Manager | Warehouse in scope |
| `inventory.post_stock_count` | Inventory Manager, Owner | Variances above cap trigger approval — see §17.3 |
| `inventory.revise_standard_cost` | Owner, Accountant | `{"max_variance_pct": 10}` beyond which Owner approval required |
| `inventory.allocate_landed_cost` | Accountant, Inventory Manager | Posted GRN required as source |
| `inventory.manage_reorder` | Inventory Manager, Purchaser | — |
| `inventory.manage_price_list` | Owner, Inventory Manager | — |
| `inventory.delete_item` | Owner | Blocked if any stock ledger history exists — deactivate instead |
| `inventory.export_data` | Owner, Inventory Manager, Accountant | — |
| `inventory.import_data` | Owner, Inventory Manager | LLM-mapped imports go through preview + approval before commit |

### 17.3 Default approval chains

| Action | Threshold / trigger | Chain |
|---|---|---|
| **Stock adjustment** | Value ≤ Stock Keeper cap | Stock Keeper posts (no approval). |
| | Value ≤ Inventory Manager cap | Stock Keeper initiates → Inventory Manager approves. |
| | Value > Inventory Manager cap | Stock Keeper initiates → Inventory Manager reviews → Owner approves. |
| **Stock count variance** | Absolute variance ≤ 1% of counted value | Inventory Manager posts. |
| | Variance > 1% | Inventory Manager reviews → Owner approves; material variances require documented cause (shrinkage, damage, process error). |
| **Warehouse transfer** | Always | Source confirms dispatch → Destination confirms receipt. No approver unless cross-branch with policy flag set. |
| **Cross-branch transfer** | Tenant policy toggle | Source Stock Keeper initiates → Source Branch Manager approves → Destination Stock Keeper receives. |
| **Standard cost revision** | Variance ≤ cap | Accountant posts. |
| | Variance > cap | Accountant proposes → Owner approves. Revaluation journal auto-generated. |
| **Landed cost allocation** | Always | Stock Keeper captures charges → Accountant allocates + posts → month-end revaluation reconciled. |
| **Item deletion (logical)** | — | Owner-only; verifies no open POs, SOs, or unreconciled ledger balance. |
| **Negative stock override** | Per-transaction toggle | Requires `inventory.override_negative_stock` permission + reason; auto-routed to Owner review on month-end report. |
| **Bulk import** | File size / row count threshold | User uploads → LLM maps columns → preview → Inventory Manager approves batch → commit with per-row audit. |

### 17.4 Scope enforcement (warehouse-level)

- `BranchWarehouseAccess` (§6.2) defines per-role read / write / transfer permissions per warehouse.
- A Stock Keeper in Warehouse A cannot view stock levels in Warehouse B unless explicitly granted.
- Transfers require **both** source write AND destination write permissions on the initiator; receipt requires destination write on the receiver.
- All queries auto-filter via RLS + warehouse scope check; UI hides out-of-scope warehouses from selectors.

### 17.5 Segregation of duties (enforced)

- **Stock counts**: the user who executes the count cannot be the user who approves the variance — auto-routed to skip-level if same.
- **Adjustments**: initiator never self-approves; system blocks approvals by `created_by`.
- **Landed cost**: captured by Stock Keeper/Purchaser, allocated and posted by Accountant — separation of capture vs posting.
- **Item master cost edits**: flagged as sensitive; change log includes before/after, user, timestamp, reason; visible in `audit_log` with `sensitive=TRUE`.

### 17.6 Audit trail

Every posting, adjustment, transfer, count, cost revision, landed cost allocation, and override writes to `audit_log` (Part 7) with action type, actor, before/after state, and reason code. Stock ledger entries themselves are immutable — corrections are always new entries (see §4.3).
