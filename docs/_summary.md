## data-model-01-foundation.md (300 lines)

**Purpose**: Multi-tenancy architecture, primary key strategy, deletion strategy, audit columns, naming conventions, timezone handling, and JSON vs relational trade-offs.

**Key sections**: Multi-tenancy pattern (shared schema + tenant_id + RLS, not schema/database-per-tenant); 6-layer defense-in-depth isolation (app-level filter, ORM scoping, RLS, connection-level, audit, automated security tests); Platform tables (no tenant_id, Super Admin only); Connection-level tenant context (SET LOCAL in middleware); Cross-tenant query prevention; UUID v7 primary keys (time-ordered, index-friendly); Hybrid deletion (soft for financial/audit-significant, hard for ephemeral); Audit columns (tenant_id, created_at/updated_at, created_by/updated_by, deleted_at/deleted_by, version for optimistic locking); Additional audit columns on financial documents (locked, locked_at, posted_at, posted_by); Trigger-maintained updated_at + version; Naming conventions (snake_case plural for tables, singular for columns, _id suffix for FK, is_/has_ prefix for booleans, _at for timestamps, _date for dates, lowercase enum values); Timezone handling (all timestamps UTC, business dates as DATE in tenant timezone, display conversion at app layer).

**Notable for implementation**: RLS policies enforce tenant isolation at database level; ORM middleware enforces tenant filter at application level — queries without tenant_id throw errors before DB; UUID v7 time-ordered for low index fragmentation on high-write tables; Stock ledger and journal_lines are immutable (never UPDATE/DELETE, corrections post new rows); Soft-deleted rows filtered by default (added_at IS NULL in all queries); Optimistic locking via version column prevents concurrent modification conflicts.

## data-model-08-performance.md (200 lines)
